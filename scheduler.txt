claude: They were abandoned because Con Kolivas got frustrated with the Linux kernel community, not because the
        schedulers were bad. He felt the kernel developers prioritized server workloads over desktop responsiveness, and
        he eventually gave up trying to get his work merged. It was a political/social outcome, not a technical one.

        His core insight — that a simpler scheduler with less bookkeeping overhead produces better interactive
        responsiveness on machines with fewer cores — was sound and is still valid.

        **What happened after BFS/MuQSS:**

        Linux replaced CFS with **EEVDF** (Earliest Eligible Virtual Deadline First) in 2023. EEVDF addresses some of
        the latency complaints that motivated BFS in the first place — it's better at ensuring interactive tasks get CPU
        time quickly. It's the current state of the art in mainline Linux.

        There's also **sched-ext** (landed in Linux 6.12, 2024), which lets you write scheduler policies as eBPF
        programs that load at runtime. This means you can experiment with scheduling algorithms without recompiling the
        kernel. Several desktop-optimized schedulers have been written as sched-ext plugins.

        **What to study:**

        - **BFS/MuQSS** — yes, study these for the core ideas (simplicity, desktop optimization, single-queue design
        choices). The source code is straightforward and well-documented.
        - **EEVDF** — study this as the current best general-purpose algorithm. It handles the fairness-vs-latency
        trade-off better than CFS did.
        - **sched-ext examples** — study these for modern thinking about what desktop users actually need from a
        scheduler.

        Don't copy any of them directly. Take the principles — favor simplicity, optimize for interactive latency, use
        priority inheritance, per-CPU queues with work stealing — and build your own. Your scheduler will be simpler
        than Linux's regardless, because you don't have 30 years of edge cases to handle.

        Define a **scheduler interface** in your kernel — a trait in Rust terms — that any scheduler must implement:

        ```
        - pick_next_task(cpu) → task
        - enqueue_task(task)
        - dequeue_task(task)
        - task_tick(task)  (called on timer interrupt)
        - balance_load()   (called periodically)
        ```

        extremely tricky and not worth the engineering effort).

        Get the OS working. Only add alternative schedulers if real users report workloads where it's insufficient.
        Scheduler swapping is a cool feature, but defining the interface upfront and implementing one scheduler behind
        it is enough. The interface makes adding alternatives later straightforward without replumbing the kernel.

claude: Yes, all of those are inherent to the design:

        **Changing priority while running:** The process is in a per-CPU run queue. Changing its priority means removing
        it from its current priority level's list and inserting it into the new priority level's list. A few pointer
        operations, trivially fast.

        **Pausing (suspending) a process:** Remove it from the run queue entirely and mark it as suspended. The
        scheduler never sees it, so it never gets CPU time. Its memory stays intact.

        **Resuming:** Put it back in the run queue at its assigned priority level. It starts getting scheduled again on
        the next scheduling cycle.
