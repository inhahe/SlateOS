##  *** CORRECTION (2026-05-30): the original premise below — "std::time is
##  broken, std issues raw Linux syscalls that hit the wrong native handler"
##  — is WRONG.  std actually WORKS on this OS.  The userspace target links
##  against the sysroot's libc.a + libstubs.a (toolchain/sysroot/lib), and
##  the `posix` crate implements the libc symbols (clock_gettime, open,
##  read, write, rename, statvfs, mkfifo, utimensat, ...) as `extern "C"`
##  functions that route to NATIVE OurOS syscalls.  So std::time calls the
##  posix `clock_gettime`, which calls syscall0(SYS_CLOCK_REALTIME=14) /
##  syscall0(SYS_CLOCK_MONOTONIC=10) — the correct native numbers — and
##  std::fs / std::net / std::process likewise route through posix.  This is
##  confirmed: nmap and stat now use std::fs/std::io/std::time/std::process
##  and build clean for x86_64-ouros.
##
##  THEREFORE the proper fix for any hand-rolled tool is simply to USE STD
##  (std::time / std::fs / std::process), not to hand-correct raw syscall
##  numbers.  The earlier per-tool number fixes (date/at/dig/rsync/scp/etc.)
##  are functionally correct but are the wrong *approach*; they should be
##  migrated to std in the broader audit below.  For the handful of POSIX
##  ops std doesn't expose (statvfs, mkfifo, utimensat) declare a thin
##  `extern "C"` binding to the posix symbol — see stat for the pattern.
##
##  ORIGINAL (now-disproven) ROOT-CAUSE ANALYSIS, kept for context:
##   1. [DISPROVEN] It was believed std (os=linux/musl) issued the *Linux*
##      clock_gettime syscall (rax=228) directly via the SYSCALL instruction,
##      and that since kernel rax=228 = SYS_PIPE_POLL, std::time returned
##      garbage.  In fact std calls the libc `clock_gettime` *symbol*, which
##      our posix crate provides and routes to the native no-arg clock
##      syscalls.  std::time works.
##   2. Independently, each hand-rolled time tool wrote a raw `syscall`
##      wrapper — with NO shared source of truth for the numbers, so they
##      each guessed differently and mostly WRONG:
##        - at, date, nmap:           SYS_CLOCK_GETTIME = 40  -> kernel 40 = SYS_PORT_READ
##        - date:                     SYS_CLOCK_SETTIME = 41  -> kernel 41 = SYS_PORT_WRITE
##        - dig, rsync, scp, screen,
##          tcpdump:                  SYS_CLOCK_MONOTONIC=30  -> kernel 30 = SYS_IRQ_REGISTER
##        - stat:                     SYS_CLOCK_GETTIME = 228 -> kernel 228 = SYS_PIPE_POLL
##        - inetd, sshd:              SYS_CLOCK_MONOTONIC=10  -> CORRECT (kernel 10)
##      They also assume a POSIX combined ABI clock_gettime(clock_id,
##      timespec*) that the kernel does NOT provide.
##
##  KERNEL CLOCK ABI (the authoritative, correct interface — number.rs):
##    SYS_CLOCK_MONOTONIC = 10  no args; returns boot-relative ns in rax (i64)
##    SYS_CLOCK_REALTIME  = 14  no args; returns wall-clock ns-since-epoch in rax
##    SYS_CLOCK_SETTIME   = 15  arg0 = target_epoch_ns (u64); 0 / -EINVAL
##    SYS_CLOCK_ADJTIME   = 16  arg0 = signed delta ns (i64 bit-pattern); 0 / -EINVAL
##  There is NO combined clock_gettime(clock_id, *ts) syscall; each clock is
##  its own no-arg syscall returning a single ns value in rax.
##
##  CONSEQUENCES OBSERVED (by reading the code; not boot-tested):
##    * date: reads time via SYS_PORT_READ(=40) and `date --set` writes via
##      SYS_PORT_WRITE(=41) — i.e. completely non-functional, and the set
##      path aims a port-I/O syscall at hardware (gated by the port-I/O cap,
##      so likely EPERM rather than real harm, but it is the wrong syscall).
##    * ntpd & hwclock: read/set the clock via /proc/time, which has NO
##      provider in procfs (no "time" arm in generate()) AND procfs is
##      read-only (no write_file).  So both cannot read or set the clock;

##    * ntpd additionally truncates all corrections to whole seconds

##    * dig/rsync/scp/screen/tcpdump: their "elapsed time" calls hit
##      SYS_IRQ_REGISTER(=30) instead of clock_monotonic.

  ROOT CAUSE (architectural, NOT a single new lint): posix/src/lib.rs has
  a crate-root `#![deny(clippy::all, clippy::pedantic)]`.  rustc applies
  crate-root attributes AFTER (and at higher precedence than) the
  command-line lint flags that Cargo derives from `[workspace.lints]`.  So
  the in-source group deny re-denies the ENTIRE pedantic group and
  overrides every per-lint `= "allow"` in the workspace `[workspace.lints.
  clippy]` block.  The only allows that survive are the ones ALSO listed in
  posix's in-source `#![allow(...)]` block (cast_possible_truncation, etc.)
  — a crate-root specific allow beats a crate-root group deny.  This is why
  unreadable_literal/must_use_candidate fire as hard errors despite being
  workspace-allowed.

  This is a genuine DESIGN CONFLICT between two standards:
    (a) CLAUDE.md mandates `#![deny(clippy::all, clippy::pedantic)]` in
        every crate (and says only the human edits CLAUDE.md), vs.
    (b) the newer workspace `[workspace.lints.clippy]` block, which sets
        `pedantic = "warn"` + a centralized allow-list and documents itself
        as the intended, "centralised and reviewable" suppression
        mechanism.
  They are mutually exclusive: as long as each crate hard-denies the
  pedantic group in source, the centralized warn+allow-list is dead code.

  RESOLUTION OPTIONS (needs a human policy call — do not flip unilaterally):
    OPT 1 (recommended, eliminates duplication): remove the redundant
      crate-root `#![deny(clippy::all, clippy::pedantic)]` from each crate
      and rely on `[lints] workspace = true`.  Then the workspace config is
      authoritative: clippy::all stays deny, pedantic is warn, and the
      allow-list actually suppresses unreadable_literal/must_use_candidate/
      etc.  Residual non-allowed lints (cast_lossless, used_underscore_items,
      doc_lazy_continuation, ptr_cast_constness, ...) then surface as
      warnings to fix or add to the workspace allow-list.  Downside:
      pedantic becomes warn-level workspace-wide, which relaxes CLAUDE.md's
      "deny in every crate" rule — hence a human policy call.
    OPT 2 (preserves deny, but duplicates): keep the crate-root deny and
      copy the full workspace allow-list into each crate's in-source
      `#![allow(...)]` block.  This is exactly the per-crate duplication the
      workspace block was created to avoid (band-aid accumulation).
  TRIGGER: dedicated lint-policy pass once the human picks an option.  Not
  blocking feature work — bare-metal build and all host tests are green.

## Known Bugs / Limitations

- (FIXED) Workqueue worker stack overflow (#DF): Task 61 ("kworker") double-
  faulted during workqueue::self_test() with RSP 0x180 bytes below stack
  bottom (33152/32768 bytes used). Debug builds have unoptimized frames.
  Fix: increased TASK_STACK_SIZE from 2 frames (32 KiB) to 4 frames (64 KiB)
  and updated mm::kstack::STACK_FRAMES to match.

- (FIXED) Init process null-deref after BOOT_OK: Was caused by the same
  BTreeMap corruption as below (use-after-free via raw pointers during
  context switch).  Fixed by Box<Task> stable-address approach.  Init
  process now runs successfully on SMP, verified in boot-test serial output.

- (FIXED) Init task BTreeMap corruption on SMP: Was a use-after-free via
  raw pointers extracted from the task BTreeMap during context switch.
  After dropping the SCHED lock, concurrent BTreeMap operations on other
  CPUs could rebalance the tree, invalidating the pointers.  Fix: changed
  `BTreeMap<TaskId, Task>` to `BTreeMap<TaskId, Box<Task>>` so tasks have
  stable heap addresses independent of BTree node layout.

- kchannel multi-task self-test crash (BINARY-LAYOUT-DEPENDENT):
  The kchannel self-test's multi-task test triggers a recursive kernel
  page fault at a kernel text address with "not-present, read, kernel"
  error.  Bisection confirmed: adding ~1,768 bytes of code anywhere in
  the kernel (e.g., a serial_println! in hotplug::enable()) shifts the
  binary layout past a critical page boundary, causing a kernel text page
  to become "not-present" during the kchannel test.  The fault handler
  itself faults on the same unmapped text, consuming stack until death.
  ROOT CAUSE: protect module's harden_kernel_sections() (kernel-core zone,
  kernel/src/mm/protect.rs ~line 488) doesn't correctly cover all kernel
  text pages at certain binary sizes.  The ELF section boundaries used to
  set page permissions leave gaps when the kernel grows past certain
  size thresholds.
  WORKAROUND: Keep hotplug::enable() minimal (no serial_println).
  FIX NEEDED (kernel-core zone): harden_kernel_sections() must compute
  kernel text boundaries from the actual loaded ELF sections (or linker
  symbols like __text_start/__text_end) and ensure ALL pages in the
  range are mapped present+executable.  Currently some pages at the end
  of the text section get marked not-present when the section spans a
  page boundary that the hardening logic doesn't account for.

- Debug kernel binary too large for Limine: The debug build (~9MB) causes
  Limine to panic "High memory allocator: Out of memory" with 128M QEMU.
  Workaround: use release builds for boot testing, or increase QEMU memory
  to 256M+.  Long-term: strip debug sections or use split-debuginfo.

  ## Judgment Calls (klogctl emulation over a non-consuming seq-cursor API):
    * The READ cursor and clear floor are PER-PROCESS atomics, not the global
      kernel state Linux's syslog uses.  Each process gets an independent view;
      a "clear" in one process does not affect another.  This is fine for the
      dmesg/journald read paths but differs from Linux's shared consuming read.
    * "Clear" is EMULATED by hiding entries at/below a stored seq floor — the
      kernel has no log-clear syscall and the ring is non-consuming, so the
      entries still physically exist (and are still visible to other processes
      / a fresh process).  PROPER FIX if true clearing is needed: add a kernel
      SYS_LOG_CLEAR (and/or per-reader consuming cursors) to klog.rs.
    * A truncated READ_ALL returns the OLDEST entries that fit (the kernel
      reads oldest->newest and stops when the buffer fills); Linux returns the
      NEWEST that fit.  Acceptable while the ring (256 entries) is small.
    * SIZE_BUFFER reports currently-buffered bytes, not the ring's fixed byte
      capacity (the kernel does not expose capacity).  This is what dmesg
      actually needs (it sizes its READ_ALL buffer from it); a tiny race exists
      if entries arrive between SIZE_BUFFER and READ_ALL.  PROPER FIX: expose
      ring capacity via SYS_LOG_READ (e.g. a query mode) if a caller needs the
      true capacity.

## Judgment Calls

- (2026-05-30) socketpair() supports only AF_UNIX + SOCK_STREAM.
  posix/src/socket.rs socketpair() is backed by the kernel stream-socket
  object (kernel/src/ipc/stream_socket.rs, SYS_SOCKETPAIR_* 300-310), a
  pair of bonded byte-stream ring buffers.  This covers the overwhelmingly
  common use of socketpair() (parent/child byte pipes, CLOEXEC fd passing
  scaffolding, libraries that want a bidirectional pipe).  Not implemented:
  * SOCK_DGRAM  → EOPNOTSUPP.  A datagram socketpair needs a message-framed
    kernel object (preserve record boundaries, each send() = one recv()),
    which the stream ring does not provide.  Proper fix when needed: add a
    datagram variant to stream_socket.rs (or a sibling dgram_socket.rs)
    that stores length-prefixed records in the ring and have socketpair()
    dispatch to it for base_type==SOCK_DGRAM; add a HandleKind::UnixDgram
    in posix with framed send/recv arms.  Trigger: a ported program that
    actually calls socketpair(AF_UNIX, SOCK_DGRAM, ...).
  * SOCK_SEQPACKET → EOPNOTSUPP.  Same framing requirement as SOCK_DGRAM
    plus reliable/ordered/connection semantics (which the bonded pair
    already gives); so SEQPACKET is "framed DGRAM over the bonded pair."
    Implement together with the DGRAM variant.
  * SCM_RIGHTS / ancillary fd passing over the pair is NOT supported — the
    stream object moves bytes only, not capability handles.  Fix when an
    fd-passing program is ported: extend the kernel object to carry an
    out-of-band handle queue alongside the byte ring and wire sendmsg/
    recvmsg's cmsg path to it.
  All three are easy to add incrementally without disturbing the existing
  stream path; socketpair() already validates base_type up front so adding
  a new arm is localised.

- (2026-05-29) POSIX signal delivery via kernel signal shim (roadmap
  "Full signal delivery via IPC", posix/src/signal.rs + kernel/src/proc/signal.rs).
  Decisions made, all easy to revisit:
  * 64 signals only.  The kernel SignalState pending/blocked masks are
    single u64s, so only signals 1..=64 are deliverable asynchronously.
    Realtime signals (SIGRTMIN=32..SIGRTMAX=64) fit, but the userspace
    SigsetT is 1024 bits; sigprocmask only mirrors bits[0] (signals
    1..64) to the kernel and sigpending only reads back the low word.
    Higher bits round-trip in userspace but never reach the kernel.
    Fix when >64 signals are actually needed: widen kernel masks to a
    bitset and the MASK/PENDING syscalls to pointer-based multi-word.
  * Auto-masking of the delivered signal during its handler: IMPLEMENTED
    (2026-05-29) entirely in userspace, no kernel/ABI change required.
    dispatch_self_signal() now (a) leaves a blocked signal pending
    instead of delivering it, and (b) before invoking a handler, unions
    the delivered signal (unless SA_NODEFER) + the handler's sa_mask into
    BLOCKED_MASK and mirrors it to the kernel via SYS_SIGNAL_MASK, then
    restores the previous mask after the handler returns.  This prevents
    re-entrancy on both paths: a synchronous re-raise finds the signal
    blocked → pending, and an async redelivery is suppressed by the
    kernel honouring the pushed mask for the handler's duration.
    SA_RESETHAND resets the disposition to SIG_DFL before the handler
    runs.  The decision logic is factored into the pure plan_self_dispatch()
    (unit-tested) so dispatch_self_signal() is a thin executor.
    Mask-restore-on-escape: CLOSED (2026-05-29).  sigsetjmp/siglongjmp
    (posix/src/setjmp.rs) now save/restore the signal mask:
    sigsetjmp(env, savemask!=0) records the current blocked mask in the
    sigjmp_buf via __posix_sigjmp_save_mask, and siglongjmp restores it via
    __posix_sigjmp_restore_mask (mirroring to the kernel) before jumping.
    So a handler that escapes its frame with siglongjmp correctly unwinds
    the mask that dispatch installed.  JmpBuf/SigjmpBuf were enlarged from
    [u64;8] to [u64;16] to hold the mask-was-saved flag (slot 8) and saved
    mask (slot 9), matching glibc's identical jmp_buf/sigjmp_buf sizing.
    Residual gap: a handler that escapes via plain longjmp (not siglongjmp)
    still skips the restore — but that is correct POSIX behaviour (longjmp
    is defined not to touch the signal mask; only siglongjmp does).
  * sigaltstack is accepted but ignored — handlers run on the normal
    stack.  The kernel builds the SignalContext at (user_rsp - ctxsize)
    & !0xF.  Honoring SS_ONSTACK would require the kernel to know the
    alt-stack base/size (another register call) and switch to it.
  * Self-directed kill()/raise() stay synchronous (dispatch_self_signal
    in-process) rather than round-tripping through SYS_SIGNAL_SEND.
    Matches existing behavior and avoids a needless kernel round-trip;
    the observable result is identical.
  * Cross-process kill() now sends ALL valid signals via SYS_SIGNAL_SEND
    and no longer classifies by default disposition on the sender side
    (the previous shim shortcut: ignore→0, stop/continue→ENOSYS).  This
    is the correct POSIX model — the kernel/target decides the action.
    The old sender-side cross-process tests were rewritten to test the
    pure kill_target() router + signal_send_errno() mapper instead.

  This note described the old SYS_FS_LIST_DIR/SYS_FS_STAT snapshot-diff

- (2026-05-27) posix getdents64 uses a per-fd snapshot cache, NOT
  incremental kernel iteration.  Each fd's first getdents64 call
  invokes SYS_FS_LIST_DIR for the whole directory; subsequent calls
  drain that snapshot.  On EOF the cache slot is freed.  Pool is
  4 slots, so at most 4 concurrent getdents64-iterated directories
  may be in progress before ENFILE is returned to new openers.
  Cache is keyed by fd number — if a program closes a directory fd
  mid-iteration and reuses the fd number, the new opener will see
  stale snapshot data until it consumes the rest of the snapshot.
  Real programs do not close-and-reuse mid-iteration, but the
  deferred proper fix is to hook fdtable::close_fd() to invalidate
  caches when their owning fd is released (mirroring how DIR_POOL
  is implicitly invalidated by closedir()).  Also: legacy getdents
  (32-bit-ino variant) remains ENOSYS — glibc/musl ship no wrapper
  for it and our 64-bit inodes don't fit the legacy struct.

## Clippy lint cleanup

UPDATE (2026-05-27): Workspace lint policy relaxed in commit 948fd5ff.
Reasoning:
- The previous `pedantic = deny` policy generated 16,959 errors in
  the kernel alone, the overwhelming majority being pure-style flags

  register pairs).  This blocked all forward work without any
