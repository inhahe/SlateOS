> Can you give me an overview of how memory allocation currently works on       
  Linux? Like how it stores chunks of used and available memory, and how it     
  finds where to allocate a new request, and whether it ever actually moves     
  used memory around.                                                           

● Here's how it works at both the userspace (glibc malloc/ptmalloc2) and kernel
  levels:

  Kernel level: virtual memory

  The kernel doesn't hand out physical RAM directly. Each process gets a virtual
   address space, and the kernel maps virtual pages (4 KiB typically) to
  physical frames on demand.

  - brk/sbrk — expands the process's contiguous heap segment (the "program
  break"). Small allocations go here.
  - mmap — maps anonymous pages anywhere in the address space. Large allocations
   (typically >=128 KiB in glibc) use this instead of growing the heap.
  - Lazy allocation — the kernel doesn't back a virtual page with physical RAM
  until the process actually writes to it (page fault). So malloc returning
  successfully doesn't mean physical memory is reserved yet.

  Userspace level: glibc ptmalloc2

  This is the allocator most Linux programs actually use. It sits between your
  code and the kernel syscalls.

  Chunk structure

  Every allocation is a chunk with a small header (typically 8–16 bytes) stored
  immediately before the returned pointer:

  | prev_size (if prev chunk is free) | size + flags (A|M|P) | user data ... |

  - size includes the header, aligned to 16 bytes (64-bit). The low bits are
  flags: PREV_INUSE, IS_MMAPPED, NON_MAIN_ARENA.
  - Free chunks reuse the user-data area to store linked-list pointers (fd, bk)
  that thread them into freelists.

  How free memory is organized (bins)

  Free chunks live in several bin structures, each optimized for different size
  ranges:

  ┌──────────┬───────────────┬────────────────────┬───────────────────────┐
  │ Bin type │     Sizes     │     Structure      │       Strategy        │
  ├──────────┼───────────────┼────────────────────┼───────────────────────┤
  │ Tcache   │ small         │ singly-linked LIFO │ fastest path, checked │
  │          │ (per-thread)  │  per size          │  first                │
  ├──────────┼───────────────┼────────────────────┼───────────────────────┤
  │ Fastbins │ ≤160 bytes    │ singly-linked LIFO │ no coalescing, speed  │
  │          │ (default)     │  per size          │ over fragmentation    │
  ├──────────┼───────────────┼────────────────────┼───────────────────────┤
  │ Small    │ < 512 bytes   │ doubly-linked      │ exact-fit lookup      │
  │ bins     │ (64-bit)      │ FIFO, exact size   │                       │
  ├──────────┼───────────────┼────────────────────┼───────────────────────┤
  │ Large    │ ≥ 512 bytes   │ doubly-linked,     │ best-fit within the   │
  │ bins     │               │ sorted by size     │ bin range             │
  ├──────────┼───────────────┼────────────────────┼───────────────────────┤
  │ Unsorted │               │ doubly-linked,     │ temporary holding     │
  │  bin     │ any size      │ single list        │ area after free,      │
  │          │               │                    │ sorted lazily         │
  ├──────────┼───────────────┼────────────────────┼───────────────────────┤
  │ Top      │ —             │ the chunk at the   │ expanded via brk when │
  │ chunk    │               │ end of the heap    │  nothing else fits    │
  └──────────┴───────────────┴────────────────────┴───────────────────────┘

  How allocation finds a chunk

  For a malloc(n) request (simplified):

  1. Tcache — check the per-thread cache for an exact-size match. If found, pop
  it. Done.
  2. Fastbin — check the fastbin for that size. Pop if available.
  3. Small bin — if the size falls in the small range, check the exact-size
  small bin.
  4. Unsorted bin — scan it. Each chunk is either used immediately (if exact
  fit) or sorted into the correct small/large bin. If a remainder is left, it
  becomes the "last remainder" chunk (helps locality).
  5. Large bins — search for a best-fit chunk (smallest chunk that's large
  enough), starting from the appropriate bin and scanning upward.
  6. Top chunk — carve off a piece from the wilderness at the end of the heap.
  7. sbrk / mmap — if the top chunk is too small, ask the kernel for more
  memory.

  Coalescing

  When a chunk is freed, ptmalloc2 checks its immediate neighbors using the
  PREV_INUSE flag and the prev_size field. Adjacent free chunks are merged into
  a single larger free chunk. This reduces fragmentation. Fastbin chunks skip
  this step initially (they're coalesced lazily when malloc needs to
  consolidate).

  Does Linux ever move used memory?

  Short answer: no, not at the virtual address level.

  - malloc / free never relocate an existing allocation behind your back. Your
  pointer stays valid until you free it.
  - realloc may move data to a new location if it can't grow the chunk in place
  — but it tells you by returning the new pointer.
  - The kernel may move the underlying physical frames (e.g., for memory
  compaction, NUMA balancing, or KSM page merging), but this is transparent —
  the virtual address doesn't change, so userspace never notices.
  - There is no compacting garbage collector. This is why long-running C
  programs can suffer from fragmentation — free space exists but is scattered in
   chunks too small to satisfy a large request.

  Why no compaction?

  C exposes raw pointers. The runtime has no way to find and update every
  pointer to a moved object, so it can't relocate live allocations. Languages
  with managed runtimes (Java, Go, C#) can do compacting GC because the runtime
  controls all references.

  ---
  If you're building your own allocator for your OS project, the key design
  decisions are: bin structure, coalescing strategy, whether to use
  per-CPU/per-thread caches, and how to handle large vs. small allocations
  differently. Happy to go deeper on any of these.

● Good questions. Let me go through each.

  1 & 2: Why exact-size only for tcache/fastbins?

  Tcache and fastbins are arrays where each slot holds a linked list of chunks
  of exactly one size class (e.g., slot 0 = 32 bytes, slot 1 = 48 bytes, slot 2
  = 64 bytes, etc.):

  tcache[0] -> [32B] -> [32B] -> NULL
  tcache[1] -> [48B] -> NULL
  tcache[2] -> NULL
  tcache[3] -> [80B] -> [80B] -> [80B] -> NULL
  ...

  You request 40 bytes, that rounds up to size class 48, you check tcache[1]. If
   it's empty, you don't check tcache[2] or tcache[3]. You just fall through to
  the next allocator stage.

  Why not check the next slot up? Two reasons:

  - Speed — the whole point of tcache/fastbins is O(1). If you start scanning
  neighboring slots, you've lost that.
  - Fragmentation — if you hand out an 80-byte chunk for a 48-byte request, you
  waste 32 bytes (too small to be useful on its own). Small-bin chunks aren't
  split because the remainder would be too tiny.

  The "find something bigger and split it" logic is deliberately deferred to the
   later stages (unsorted bin, large bins, top chunk) where the overhead is
  acceptable and the chunks are big enough to split usefully.

  4: The unsorted bin

  Yes, it's lazily sorted

  When you free() a chunk, it goes into the unsorted bin. That's it — no
  sorting, no bin placement. Fast.

  The sorting happens the next time malloc scans the unsorted bin because it
  couldn't satisfy the request from tcache/fastbins/small bins. During that
  scan, for each chunk it pulls off:

  - If it's an exact fit → return it to the caller, done.
  - If it's not a fit → place it into the correct small or large bin. Then
  continue scanning.

  So the unsorted bin empties out gradually as a side effect of malloc calls.

  What's a "remainder"?

  If during the unsorted bin scan, malloc finds a chunk that's bigger than 
  needed and big enough to split, it does this:

  Before:  [ -------- 512 bytes -------- ]  (in unsorted bin)
  Request: 80 bytes

  After:   [ 80B returned ] [ 432B "last remainder" put back in unsorted bin ]

  The leftover piece stays in the unsorted bin as the "last remainder." On the
  next malloc, ptmalloc2 checks this remainder first, which gives good locality
  — sequential allocations end up adjacent in memory.

  Unsorted bin vs. top chunk

  - Unsorted bin — holds recently freed chunks scattered throughout the heap.
  Temporary staging area.
  - Top chunk — the single chunk at the very end of the heap, bordering
  unallocated address space. It's special because it can grow via sbrk(). It's
  the allocator's last resort before asking the kernel for more memory.

  Heap layout:
  [ used ][ free ][ used ][ free ][ used ][ TOP CHUNK ~~~> brk can extend this
  way ]
                    ^                        ^
            unsorted bin chunks         only one, at the end

  Your data structure ideas

  You're actually reinventing optimizations that real allocators use — which
  means your instincts are good.

  "One list per size bin, update on alloc/free"

  This is exactly what small bins are. 62 bins, one per size class,
  doubly-linked. Insertion and removal are O(1) because you don't care about
  ordering within an exact-size bin (all chunks are the same size).

  "When a bin is empty, point to the next non-empty bin"

  ptmalloc2 solves this with a binmap — a bitmap where bit i indicates whether
  bin i is non-empty. To find the next non-empty bin ≥ size n, it does a bitwise
   scan from position n upward, which is a few CPU instructions (x86 bsf/tzcnt).
   So it doesn't linearly walk empty bins.

  "Newton's method on a sorted list" / "keep lists sorted by size"

  For large bins, ptmalloc2 does keep chunks sorted by descending size within
  each bin. But each large bin covers a range of sizes (e.g., 512–576, 576–640,
  etc.), so you need to search within the bin. ptmalloc2 does a linear scan
  within the bin, which is simple but not optimal.

  More modern allocators improve on this:

  - dlmalloc (the ancestor of ptmalloc2) uses a treap (tree + heap) — a balanced
   binary tree of free chunks keyed by size. Finding best-fit is O(log n).
  - jemalloc (used by FreeBSD, Firefox, previously by Facebook) uses red-black 
  trees (called "extent trees") for large allocations — O(log n) best-fit.
  - mimalloc (Microsoft) uses size-segregated pages — each page only holds one
  size class, so there's no searching at all for small allocations.

  Binary search (which is what "Newton's method on a sorted list" converges to)
  works on arrays but not linked lists, since you can't jump to the middle of a
  linked list in O(1). So you'd need a tree or a skip list, which is exactly
  what dlmalloc/jemalloc do.

  ---
  If you're designing an allocator for your OS, the state of the art is roughly:
   per-CPU caches of exact-size slabs for small objects (like Linux's SLUB) + a 
  tree-based best-fit allocator for large objects + buddy allocator at the 
  physical page level. Want me to go into any of these?

> Regarding the bitwise scan, isn't my idea faster? Of just storing the next    
  larger available chunk if the bin for that size is empty. Regarding the lazy  
  sorting, wouldn't it be better to do this periodically in a backround process 
   rather than when malloc runs, so that it doesn't make malloc slower?         

● Bitwise scan vs. direct pointer to next non-empty bin

  Your idea is faster for the lookup — O(1) dereference vs. a bit scan
  instruction. But the cost moves to maintenance:

  When you free a chunk into an empty bin, you now need to update every smaller
  bin that was pointing past this bin to point to it instead. Worst case, that's
   updating dozens of pointers. And when you allocate the last chunk from a bin
  (making it empty), you need to find the next non-empty bin to redirect your
  pointer — which is the same scan you were trying to avoid.

  The bitmap approach costs maybe 1-3 instructions for the scan (tzcnt on a
  64-bit word covers 64 bins at once), and updates are always O(1) — flip one
  bit on free, flip one bit when a bin empties. It's hard to beat in practice.

  That said, a hybrid could work: maintain the bitmap and cache a "next
  non-empty" pointer that you lazily invalidate. But the added complexity
  probably isn't worth it when the bitmap scan is already so fast.

  Background sorting of the unsorted bin

  This is a reasonable idea, but there are practical problems:

  Concurrency cost. The unsorted bin is protected by a mutex (the arena lock). A
   background thread sorting it would hold that lock, blocking any thread trying
   to malloc or free during that time. You'd be trading occasional latency
  during malloc for unpredictable latency when the background sorter happens to
  hold the lock.

  It solves a small problem. The unsorted bin is typically short — most chunks
  get sorted into bins within a few malloc calls. The lazy approach means
  sorting work is amortized across the allocations that benefit from it. A
  background thread adds a whole thread, synchronization, scheduling overhead,
  and cache pollution for a list that's usually near-empty.

  Cache locality. When malloc sorts the unsorted bin, it's already running on
  the core that needs the data — the bin metadata is hot in that core's cache. A
   background thread on a different core would pull that data into its cache,
  evicting it from the core that actually needs it. The next malloc then takes a
   cache miss to reload it.

  The general principle here: for something that's accessed frequently and
  briefly, doing small amounts of work inline at the point of use almost always
  beats doing batch work on a separate thread, because you avoid synchronization
   and you keep data in the right cache. Background processing shines when the
  work is large, infrequent, and independent — like garbage collection of a
  large heap, not sorting a short linked list.

● Good question, but first a clarification — there aren't just three bins. There
   are three categories of bins, each containing many individual bins:

  ┌──────────┬────────┬────────────────────────────────────────────────────┐
  │ Category │ # of   │                 Size class spacing                 │
  │          │  bins  │                                                    │
  ├──────────┼────────┼────────────────────────────────────────────────────┤
  │ Fastbins │ ~10    │ exact, every 16 bytes (32, 48, 64, ... 160)        │
  ├──────────┼────────┼────────────────────────────────────────────────────┤
  │ Small    │ ~62    │ exact, every 16 bytes (32, 48, ... 512)            │
  │ bins     │        │                                                    │
  ├──────────┼────────┼────────────────────────────────────────────────────┤
  │ Large    │        │ ranges that widen logarithmically (512–576,        │
  │ bins     │ ~63    │ 576–640, ... then 64-wide, 512-wide, 4096-wide,    │
  │          │        │ etc.)                                              │
  └──────────┴────────┴────────────────────────────────────────────────────┘

  So the real tuning parameters are:

  - Fastbin max — currently 160 bytes (was 64 for years, then 128, then 160)
  - Small/large boundary — 512 bytes on 64-bit
  - Large bin range progression — how quickly the bin width grows
  - Tcache count — 7 chunks cached per size class per thread
  - Tcache max size — 1032 bytes
  - mmap threshold — 128 KiB (above this, bypass the heap entirely)

  How much have these been tested?

  Honestly, less rigorously than you'd expect. The history:

  Doug Lea set most of these values in dlmalloc in the mid-1990s, based on
  empirical analysis of allocation patterns in programs of that era. Wolfram 
  Gloger adapted it into ptmalloc2 (adding arenas for threading), and glibc
  adopted it. The fundamental bin structure and boundaries haven't changed much
  since.

  Tcache was added in glibc 2.26 (2017) — that was the biggest structural change
   in decades, and its parameters (7 entries, ~1 KiB max) were chosen somewhat
  conservatively.

  The tuning that does happen is driven by specific real-world regressions —
  someone benchmarks MySQL or Python or Chrome, finds malloc is a bottleneck,
  and a specific parameter gets adjusted. It's not systematic sweep-testing.

  Why haven't they been swept more aggressively?

  Stability over optimality. glibc ships in every Linux distro. Changing
  allocator behavior risks breaking things that accidentally depend on
  allocation patterns (fragmentation characteristics, address layout, etc.). The
   attitude is "good enough for the general case, use mallopt() or a different
  allocator if you need better."

  No single optimal configuration exists. The ideal bin sizes depend heavily on
  the workload. A web server, a database, a compiler, and a game all have wildly
   different allocation profiles. Any fixed set of boundaries is a compromise.

  Modern allocators explored this more

  jemalloc and mimalloc essentially asked your question and came up with a
  different answer: instead of a few categories with different spacing rules,
  use a single geometric size class progression across the entire range:

  jemalloc size classes (64-bit):
  8, 16, 32, 48, 64, 80, 96, 112, 128,
  160, 192, 224, 256,          <- 25% spacing
  320, 384, 448, 512,          <- 25% spacing
  640, 768, 896, 1024,         <- 25% spacing
  ...continues doubling the step at each power of 2

  The idea: keep internal fragmentation (wasted space from rounding up) bounded
  to ~25% at every size, with no sharp behavioral transitions at arbitrary
  boundaries. This was tested more systematically — the jemalloc and mimalloc
  papers both include benchmark suites across dozens of real applications.

  mimalloc's paper (2019) specifically benchmarked against glibc malloc,
  jemalloc, tcmalloc, Hoard, and others across ~15 real programs. Their
  conclusion was that the size class spacing mattered less than page-level 
  organization (how you lay out same-size allocations in contiguous pages for
  cache and TLB efficiency).

  The honest answer

  The bin boundaries in glibc are historical artifacts, lightly tuned over 30 
  years, good enough for general use, but not optimal for any specific workload.
   The more modern allocators got better results not primarily by finding better
   bin boundaries, but by rethinking the overall architecture — per-CPU caches,
  size-segregated pages, and geometric size classes that eliminate the somewhat
  arbitrary category boundaries entirely.

  If you're designing an allocator for your OS, the geometric size class
  approach (jemalloc/mimalloc style) is probably the better starting point than
  glibc's three-category system.

> Okay, so what are the tuning parameters for the geometric size class approach 
   without separate bins? I guess just 1. the multiplication factor for the     
  next highest size?                                                            

● Roughly, yes, but there are a few more. And geometric size class allocators
  still use bins — one bin per size class. What they eliminate is the separate
  categories with different rules (fastbin vs small vs large). Every size class
  works the same way.

  The core parameters:

  1. Size classes per doubling (controls your multiplication factor)

  This is the big one. jemalloc uses 4 size classes per power of two, which
  gives roughly a 1.25x factor:

  256, 320, 384, 448, 512, 640, 768, 896, 1024 ...
       1.25x 1.2x 1.17x 1.14x 1.25x ...

  It's not a clean geometric series — they round to nice alignment boundaries,
  so the factor wobbles. But the worst-case internal fragmentation is bounded by
   1/classes_per_doubling. With 4 per doubling, you waste at most ~25%. With 8
  per doubling you waste at most ~12.5%, but you have twice as many bins to
  manage.

  2. Minimum size / alignment — typically 16 bytes on 64-bit. Everything rounds
  up to at least this.

  3. Slab page size — each size class gets memory from the OS in chunks called
  slabs (e.g., 4 KiB, 64 KiB, or 2 MiB). A slab for 64-byte objects would hold
  64 of them in a 4 KiB page. Larger slabs mean less OS interaction but more
  wasted space if the slab isn't full. Some allocators use different slab sizes
  for different size classes.

  4. Large object threshold — above some size (e.g., 32 KiB in mimalloc, 14 KiB
  in jemalloc), objects get their own dedicated pages rather than sharing a
  slab. At that point, the size class rounding still applies, but the allocator
  is basically just rounding up to a page multiple.

  5. Thread cache depth — how many freed objects to cache per size class per
  thread before returning them to the global pool. jemalloc uses a dynamic
  scheme that adjusts based on allocation rate. mimalloc uses per-thread heaps
  so this is less of a parameter.

  ---
  So the fundamental trade-off controlled by parameter 1 is:

  - Fewer classes per doubling (say 2, giving 1.41x factor) → fewer bins, less
  metadata, simpler, but up to ~41% wasted space per allocation
  - More classes per doubling (say 8, giving 1.09x factor) → less wasted space,
  but more bins means more cold cache lines and more partially-filled slabs
  sitting around

  4 per doubling (~25% max waste) is where most modern allocators landed. It
  hasn't been exhaustively proven optimal, but independent teams (jemalloc,
  mimalloc, tcmalloc) all converged on roughly the same answer, which is a
  reasonable signal.
