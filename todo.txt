  stable heap addresses independent of BTree node layout.

  The kchannel self-test's multi-task test triggers a recursive kernel
  page fault at a kernel text address with "not-present, read, kernel"
  error.  Bisection confirmed: adding ~1,768 bytes of code anywhere in
  the kernel (e.g., a serial_println! in hotplug::enable()) shifts the
  binary layout past a critical page boundary, causing a kernel text page
  to become "not-present" during the kchannel test.  The fault handler
  itself faults on the same unmapped text, consuming stack until death.
  ROOT CAUSE: protect module's harden_kernel_sections() (kernel-core zone,
  kernel/src/mm/protect.rs ~line 488) doesn't correctly cover all kernel
  text pages at certain binary sizes.  The ELF section boundaries used to
  set page permissions leave gaps when the kernel grows past certain
  size thresholds.
  WORKAROUND: Keep hotplug::enable() minimal (no serial_println).
  FIX NEEDED (kernel-core zone): harden_kernel_sections() must compute
  kernel text boundaries from the actual loaded ELF sections (or linker
  symbols like __text_start/__text_end) and ensure ALL pages in the
  range are mapped present+executable.  Currently some pages at the end
  of the text section get marked not-present when the section spans a
  page boundary that the hardening logic doesn't account for.

## Judgment Calls

- (2026-05-29) POSIX signal delivery via kernel signal shim (roadmap
  "Full signal delivery via IPC", posix/src/signal.rs + kernel/src/proc/signal.rs).
  Decisions made, all easy to revisit:
  * 64 signals only.  The kernel SignalState pending/blocked masks are
    single u64s, so only signals 1..=64 are deliverable asynchronously.
    Realtime signals (SIGRTMIN=32..SIGRTMAX=64) fit, but the userspace
    SigsetT is 1024 bits; sigprocmask only mirrors bits[0] (signals
    1..64) to the kernel and sigpending only reads back the low word.
    Higher bits round-trip in userspace but never reach the kernel.
    Fix when >64 signals are actually needed: widen kernel masks to a
    bitset and the MASK/PENDING syscalls to pointer-based multi-word.
  * Auto-masking of the delivered signal during its handler: IMPLEMENTED
    (2026-05-29) entirely in userspace, no kernel/ABI change required.
    dispatch_self_signal() now (a) leaves a blocked signal pending
    instead of delivering it, and (b) before invoking a handler, unions
    the delivered signal (unless SA_NODEFER) + the handler's sa_mask into
    BLOCKED_MASK and mirrors it to the kernel via SYS_SIGNAL_MASK, then
    restores the previous mask after the handler returns.  This prevents
    re-entrancy on both paths: a synchronous re-raise finds the signal
    blocked → pending, and an async redelivery is suppressed by the
    kernel honouring the pushed mask for the handler's duration.
    SA_RESETHAND resets the disposition to SIG_DFL before the handler
    runs.  The decision logic is factored into the pure plan_self_dispatch()
    (unit-tested) so dispatch_self_signal() is a thin executor.
    Mask-restore-on-escape: CLOSED (2026-05-29).  sigsetjmp/siglongjmp
    (posix/src/setjmp.rs) now save/restore the signal mask:
    sigsetjmp(env, savemask!=0) records the current blocked mask in the
    sigjmp_buf via __posix_sigjmp_save_mask, and siglongjmp restores it via
    __posix_sigjmp_restore_mask (mirroring to the kernel) before jumping.
    So a handler that escapes its frame with siglongjmp correctly unwinds
    the mask that dispatch installed.  JmpBuf/SigjmpBuf were enlarged from
    [u64;8] to [u64;16] to hold the mask-was-saved flag (slot 8) and saved
    mask (slot 9), matching glibc's identical jmp_buf/sigjmp_buf sizing.
    Residual gap: a handler that escapes via plain longjmp (not siglongjmp)
    still skips the restore — but that is correct POSIX behaviour (longjmp
    is defined not to touch the signal mask; only siglongjmp does).
  * sigaltstack is accepted but ignored — handlers run on the normal
    stack.  The kernel builds the SignalContext at (user_rsp - ctxsize)
    & !0xF.  Honoring SS_ONSTACK would require the kernel to know the
    alt-stack base/size (another register call) and switch to it.
  * Self-directed kill()/raise() stay synchronous (dispatch_self_signal
    in-process) rather than round-tripping through SYS_SIGNAL_SEND.
    Matches existing behavior and avoids a needless kernel round-trip;
    the observable result is identical.
  * Cross-process kill() now sends ALL valid signals via SYS_SIGNAL_SEND
    and no longer classifies by default disposition on the sender side
    (the previous shim shortcut: ignore→0, stop/continue→ENOSYS).  This
    is the correct POSIX model — the kernel/target decides the action.
    The old sender-side cross-process tests were rewritten to test the
    pure kill_target() router + signal_send_errno() mapper instead.

- (2026-05-26) Linux eventfd(): EFD_SEMAPHORE flag is unsupported.
  [DONE 2026-05-27] Kernel EventFd grew a `semaphore` field and a new

- (2026-05-26) eventfd fd inheritance across spawn is pass-through:
  closing the eventfd handle from parent OR child invalidates it for
  both, because there is no kernel-level eventfd::dup() with ref-
  counting yet.  Same caveat as pipes.  Fix: add an Arc-like
  refcount inside the EVENTFD_TABLE entry plus an eventfd::dup()
  helper, then update kernel/src/proc/spawn.rs's fd_handle_type::
  EVENTFD branch to call it (mirrors what kernel-ipc owes pipes).
  [DONE 2026-05-27] EventFd grew a `refcount: u32` field (init 1).
  `eventfd::dup(handle)` increments it and returns the same id.
  `close()` decrements and only tears down when refcount hits 0.
  spawn.rs EVENTFD branch now calls dup() instead of pass-through.
  pcb::destroy_process_resources() closes unclaimed eventfd
  initial_fds (previously fell into the FILE branch which silently
  no-op'd).  Self-test `test_dup_refcount` covers dup+partial close+
  final close+post-close dup-failure.

- (2026-05-27) posix inotify is polling-based, not event-driven.
  posix::epoll provides functional inotify_init/init1/add_watch/rm_watch
  by snapshotting directory listings (SYS_FS_LIST_DIR) and file stats
  (SYS_FS_STAT) on every read() call, diffing against the previous
  snapshot.  This means:
    * Events derivable from polling work: IN_CREATE, IN_DELETE,
      IN_MODIFY (file size change), IN_DELETE_SELF, IN_IGNORED.
    * Events that require kernel hooks are silently dropped from the
      mask: IN_OPEN, IN_ACCESS, IN_CLOSE_WRITE, IN_CLOSE_NOWRITE,
      IN_MOVED_FROM, IN_MOVED_TO, IN_MOVE_SELF, and precise IN_ATTRIB
      (we use size-change as a proxy for MODIFY but cannot detect
      pure-attribute changes like chmod/chown).
    * Move events appear as DELETE on the source watch and CREATE on
      the destination watch, not as MOVED_FROM/MOVED_TO with matching
      cookies.
    * Detection latency = read() call frequency × poll_all_watches
      cost.  No background notification; readers must call read()
      (or poll/select/epoll on the fd) actively.
    * No coalescing — burst changes between two reads collapse to
      the net diff, not the full sequence.
  Proper fix: add SYS_FS_NOTIFY_* kernel syscalls that hook into the
  VFS layer's existing fnotify (kernel/src/fs/fnotify.rs already
  exists per roadmap line 1019).  Until then, polling is acceptable
  for inotifywait-style use cases and most file watcher daemons.
  Limits: MAX_INOTIFY_INSTANCES=4, MAX_INOTIFY_WATCHES=8 per instance,
  MAX_INOTIFY_CHILDREN=16 per directory watch, MAX_INOTIFY_EVENTS=32
  queued events, INOTIFY_NAME_MAX=64, INOTIFY_PATH_MAX=256.

- (2026-05-27) posix getdents64 uses a per-fd snapshot cache, NOT
  incremental kernel iteration.  Each fd's first getdents64 call
  invokes SYS_FS_LIST_DIR for the whole directory; subsequent calls
  drain that snapshot.  On EOF the cache slot is freed.  Pool is
  4 slots, so at most 4 concurrent getdents64-iterated directories
  may be in progress before ENFILE is returned to new openers.
  Cache is keyed by fd number — if a program closes a directory fd
  mid-iteration and reuses the fd number, the new opener will see
  stale snapshot data until it consumes the rest of the snapshot.
  Real programs do not close-and-reuse mid-iteration, but the
  deferred proper fix is to hook fdtable::close_fd() to invalidate
  caches when their owning fd is released (mirroring how DIR_POOL
  is implicitly invalidated by closedir()).  Also: legacy getdents
  (32-bit-ino variant) remains ENOSYS — glibc/musl ship no wrapper
  for it and our 64-bit inodes don't fit the legacy struct.

## Clippy lint cleanup

UPDATE (2026-05-27): Workspace lint policy relaxed in commit 948fd5ff.
Reasoning:
- The previous `pedantic = deny` policy generated 16,959 errors in
  the kernel alone, the overwhelming majority being pure-style flags
  (doc backticks, format-arg inlining, similar names from hardware
  register pairs).  This blocked all forward work without any
  correctness benefit.
- The new policy keeps `clippy::all = deny` and lowers `clippy::
  pedantic` to warn, with explicit `allow` entries for ~28 specific
  pure-style pedantic lints — each with an inline comment in
  Cargo.toml explaining why it's not worth enforcing.
- Correctness-relevant lints (unwrap_used, expect_used, panic,
  indexing_slicing, arithmetic_side_effects) remain at warn level.
  These represent ~16,800 warnings in the kernel that need focused
  follow-up:
    * ~7,500 indexing_slicing — convert to `.get()` / `.get_mut()`
      where the index could overflow, or document why the index is
      proven in-range with a comment.
    * ~5,700 arithmetic_side_effects — convert to `checked_*`,
      `wrapping_*`, or `saturating_*` per intent.
    * ~3,000 expect_used + unwrap_used — convert to `?` or
      explicit error handling; keep only those that are truly
      infallible (with a justifying comment).
- ~2,100 remaining pedantic warnings (cast_lossless, manual_div_ceil,
  map_or, unnecessary_wraps, etc.) are mechanical improvements that
  `cargo clippy --fix` declines to apply (reports "the suggestion
  might not be applicable in constant blocks" and aborts).  These
  need manual cleanup, ideally module-by-module.

Tackle as a focused effort in a dedicated session — pick one lint
category, fix all instances across a subsystem, commit, repeat.
Do NOT gate other roadmap work on this.
