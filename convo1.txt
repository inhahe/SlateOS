claude: **Queryable metadata vs WinFS:**

        Similar concept but different scope. WinFS was trying to replace the filesystem entirely with a relational
        database — every file was a database row with a full schema, relationships between files, SQL queries, etc. It
        was so ambitious that Microsoft couldn't ship it.

        BeOS's BFS was much simpler: files have normal filenames in normal directories, but you can attach arbitrary
        typed attributes to any file (artist, bitrate, email_sender, whatever), and the filesystem indexes those
        attributes so queries are fast. It's less "the filesystem is a database" and more "the filesystem has a search
        index built in."

        For your OS, the BFS approach is practical and WinFS is not. Let files live in a normal hierarchy, but allow
        indexed attributes that programs can set and users can query. Your file explorer's column view (lines 461-481 in
        design.txt) already describes exactly this — showing bitrate, resolution, ID3 tags, EXIF data as columns. Having
        the filesystem index those attributes makes searching instant instead of having to open and parse every file.

        **Per-process namespaces:**

        Not an API detail — it's about what a process *sees* when it accesses the filesystem. Different processes see
        different filesystem trees.

        Example: you run a sandboxed browser. From the browser's perspective, `/home` contains only
        `/home/user/Downloads`. It can't see other users' home directories or even other directories in your own home.
        But from the terminal's perspective, `/home` shows everything. Both processes use the same `open()` syscall —
        they just see different contents at the same paths.

        This is how Linux containers (Docker) work. A containerized process thinks it has its own root filesystem, its
        own `/etc`, its own `/tmp`. In reality, the kernel is showing it a customized view. The process doesn't opt into
        this — its parent sets up the namespace before launching it.

        For your OS, this is useful for sandboxing. Instead of relying solely on capability checks ("can this process
        open this file?"), you can make files the process shouldn't access simply not exist in its view of the
        filesystem. The browser doesn't need to be denied access to `/etc/shadow` — it lives in a namespace where
        `/etc/shadow` doesn't exist.

        **Factotum:**

        It's Plan 9's credential manager. On most OSes, every program that needs to authenticate handles passwords/keys
        itself — your SSH client stores your SSH key, your browser stores website passwords, your email client stores
        your email password. Each program implements its own credential storage, its own password prompting, its own
        encryption for stored secrets.

        Factotum centralizes this. It's a single process that holds all credentials. When SSH needs to authenticate, it
        doesn't read your private key — it asks factotum "please sign this challenge." Factotum signs it and returns the
        result. SSH never sees the key. When a website needs a password, the browser asks factotum, factotum prompts the
        user (or retrieves the stored password), and hands it to the browser.

        Benefits:
        - Programs never touch raw credentials, so a compromised program can't steal your passwords
        - One place to manage all credentials
        - One place to implement encryption, secure storage, and user prompting correctly
        - Adding a new auth method (hardware key, biometric) only requires updating factotum, not every program

        This maps directly to your design. You mentioned (line 370) an API for username/password fields and OS-level
        credential autofill. A factotum-like service is exactly the right architecture for that. Programs call the
        credential service, the service handles storage and user verification.

        **UTF-8 origin:**

        I just meant that UTF-8 was invented by Ken Thompson and Rob Pike while working on Plan 9. It was designed
        specifically for Plan 9 and then adopted by the rest of the world. I listed it as a "good idea from Plan 9"
        because it literally came from Plan 9. Not something you need to act on — you'll obviously use UTF-8 regardless.
        It was a historical note.

claude: **BFS vs ext4:**

        I think there's a name collision confusing things. BFS (Be File System) from BeOS is a *filesystem* — it
        competes with ext4. BFS (Brain Fuck Scheduler) by Con Kolivas is a *process scheduler* — completely unrelated,
        competes with CFS/EEVDF.

        Nobody uses Be's BFS because BeOS died in 2001. The company went bankrupt and the OS stopped being developed.
        Haiku OS (the open-source successor) continues BFS development, but it's a tiny niche project. ext4 has 20 years
        of battle-testing, thousands of contributors, and runs on billions of devices. BFS had some great ideas
        (queryable attributes) but never had the chance to mature to that level of reliability. The feature gap isn't
        about quality — it's about one project having massive ongoing investment and the other not.

        You can adopt BFS's good ideas (indexed queryable attributes) in whatever filesystem you use. You don't need to
        use BFS itself.

        **File extensions for executables and libraries:**

        Changing gcc/llvm's *output* extension is trivial — you just specify it with `-o`:

        ```
        gcc main.c -o main.run          # produces main.run instead of a.out
        gcc -shared lib.c -o lib.dso    # produces lib.dso instead of lib.so
        ar rcs lib.slib *.o             # produces lib.slib instead of lib.a
        ```

        The harder part is the **linker search rules**. When you write `gcc -lfoo`, the linker looks for `libfoo.so` and
        `libfoo.a` in its search paths. That's hardcoded in the linker (ld/lld). Changing this requires either:

        - **Linker configuration** — lld (LLVM's linker) supports custom library search suffixes. You'd configure it
        once for your OS's toolchain. This is a small patch or config change, not a major modification.
        - **Build system configuration** — cmake, meson, pkg-config all assume .so/.a. Each needs a platform definition
        file that says "on this OS, shared libraries are .dso and static libraries are .slib." Both cmake and meson
        support this — they already have different rules for macOS (.dylib), Windows (.dll/.lib), and Linux (.so/.a).
        Adding yours is a configuration file, not a code change.

        So it's not hard — it's a one-time toolchain setup, not ongoing friction.

        **My revised recommendations:**

        Your reasoning is sound. These libraries won't be binary-compatible with Linux, so using .so/.a would be
        misleading. Pick extensions that are:
        - Distinct from Linux and Windows
        - Self-documenting
        - Short

        Some options:

        | Type | Option A | Option B | Option C |
        |---|---|---|---|
        | Executable | .run | .exe (just take it, it's not patented) | .bin |
        | Shared library | .dso | .shl | .lib |
        | Static library | .slib | .sta | .ar |

        .exe is honestly fine. It's universally understood. Your executables won't be Windows-compatible regardless —
        the extension just communicates "this is an executable." If that feels too Windows-associated, .run is clean and
        obvious.

        For shared libraries, I'd lean toward .dso (dynamic shared object) — descriptive, not taken by any major
        platform, short.

        For static libraries, .slib (static library) — same reasoning.

        Whatever you pick, the toolchain configuration is a few files. Not a significant effort.
