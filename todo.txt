## Known Bugs / Limitations

- Debug kernel binary too large for Limine: The debug build (~9MB) causes
  Limine to panic "High memory allocator: Out of memory" with 128M QEMU.
  Workaround: use release builds for boot testing, or increase QEMU memory
  to 256M+.  Long-term: strip debug sections or use split-debuginfo.

- PI futex: transitive priority inheritance — scheduler-side infrastructure
  is DONE (blocked_on_pi_addr field, pi_chain_boost(), cycle detection,
  depth limit, tested).  Waiting on kernel-ipc zone to wire it up in
  futex_lock_pi/futex_unlock_pi (see cross-zone request below).

## Cross-Zone Requests

- kernel-ipc: Wire SYS_CLOCK_MONOTONIC (syscall/handlers.rs) to use
  hpet::elapsed_ns() when HPET is available, falling back to the APIC tick
  counter if not.  The HPET gives 10 ns resolution vs 10 ms from the APIC
  timer.  hpet::is_available() and hpet::elapsed_ns() are the public API.

- kernel-ipc: Wire SYS_SLEEP (syscall/handlers.rs) to use HPET-based
  timing for sub-10ms sleep accuracy.  Currently rounds up to 10 ms ticks.

- kernel-ipc: Store exit code from SYS_EXIT (handlers.rs:131) in PCB
  state so SYS_PROCESS_WAIT can retrieve it.  Currently the exit code
  argument is accepted but discarded.

- kernel-ipc: Fix mmap partial failure recovery (handlers.rs:500, 548).
  If mmap region partially maps then alloc_frame fails, already-mapped
  frames leak.  Need to track and unmap on error.

- kernel-ipc: Verify SYS_PROCESS_ID (handlers.rs:1549) returns the
  owning process PID, not just the task ID.  Has a TODO comment.

- kernel-ipc: Wire transitive priority inheritance into futex_lock_pi /
  futex_unlock_pi (ipc/futex.rs).  The scheduler-side infrastructure is
  ready in sched/mod.rs and sched/task.rs:
    - `Task.blocked_on_pi_addr: Option<u64>` �� set before blocking, clear on acquire
    - `sched::set_blocked_on_pi_addr(task_id, addr)` — setter
    - `sched::get_blocked_on_pi_addr(task_id)` — getter
    - `sched::pi_chain_boost(start_owner, donor_prio, find_owner)` — walk chain
    - `task::PI_CHAIN_DEPTH_LIMIT` (10) — max chain depth
  In futex_lock_pi: after `sched::boost_priority(owner, caller_prio)`,
  call `sched::pi_chain_boost(owner, caller_prio, |addr| pi_table.find_owner(addr))`.
  Before blocking: `sched::set_blocked_on_pi_addr(caller_task, Some(futex_addr))`.
  After acquiring: `sched::set_blocked_on_pi_addr(caller_task, None)`.
  Self-test already passes for the sched-side; needs futex-side integration test.

- fs: Need a ramfs/tmpfs implementation for the root mount point.
  Currently no filesystem is mounted at "/" at boot, so init's writes
  to /bin/hello and /bin/ticker fail with NotFound.  A simple in-memory
  filesystem would unblock init binary deployment.  Alternatively,
  mount the ESP's FAT filesystem at boot.

## Judgment Calls

- boot-test.ps1 and the copy-to-ESP step: Limine loads the kernel from
  esp/boot/kernel (per limine.conf kernel_path: boot():/boot/kernel).  The
  esp/EFI/BOOT/ directory is for the bootloader EFI binary, not the kernel.
  Always copy built kernel to esp/boot/kernel.
