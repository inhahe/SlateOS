Have an alloc function where the program can allocate X amount of memory and it won't fail as long as the user has that much memory plus space on the filesystem, because the OS swaps the rest? (lazily on write)

"
PUSHBACK: This is what Linux already does by default (overcommit). The problem
is well-documented: programs allocate more virtual memory than physical RAM +
swap can back, everything works fine until programs actually use all that
memory, and then the OOM killer starts killing processes to free memory. The
promise of "won't fail" is a lie — it fails later, catastrophically, instead of
failing early, gracefully.

Better approach: let programs request "committed" memory (guaranteed to be
backed by physical RAM + swap) or "lazy" memory (may fail on access if
overcommitted). Default to committed for reliability. Programs that know what
they're doing (like databases that manage their own memory) can opt into lazy.
"

PARTIAL PUSHBACK: Automatically growing stacks via page fault handlers is a
real technique (Go does this, for example). But "won't run out" is dangerous —

 Atomic updates and rollback:
  This is where newer package managers shine. If an update breaks something:
  - Traditional (apt, pacman) — you're partially updated, rolling back means manually downgrading packages. Messy.
  - Atomic (Nix, OSTree/Silverblue) — the entire system state is versioned. An update either fully applies or doesn't.
  Rolling back is switching a pointer to

 Possible middle ground for your OS: Use content-addressed packages for isolation and rollback, but support dynamic
  linking within a generation. Within one system generation, apps share .so files. A security update rebuilds the shared
   library and creates a new generation where apps point to the new .so. You get fast security patches (rebuild one
  library, not every dependent) with atomic rollback (switch generation pointer). The cost: you lose Nix's guarantee
  that each package is fully self-describing, and you reintroduce the possibility of a shared library update breaking an
   app (though rollback mitigates that).

Nix has an exception mode where it stores libraries in the standard accepted directories, specifically for binary program installations, so security fixes can be done for those too

Whenever Nix updates a library by making a whole new instance, it has to go through every program that uses it and change the source and recompile it. There must be a better way: have a middleman somewhere that the source can refer to and the middleman points to the latest update (I guess this change means we have to use dynamic link libraries?).    The chain would be:

        ```
        Windows app
            ↓ calls Windows API (CreateFile, etc.)
        WINE (translates to POSIX/Linux calls)
            ↓ calls Linux syscalls (open, read, etc.)
        Your Linux syscall translation layer
        ```

        POSIX:

        - Linux-specific ioctls for graphics (DRM/KMS for GPU access)

        - Linux-specific socket options
        - Sound subsystem (ALSA/PulseAudio)

        The basic POSIX layer gets you maybe 60% of the way. The remaining 40% is Linux-specific interfaces that WINE
        depends on. But you'd be implementing those in your compatibility layer anyway if you want to run other Linux
        software, so WINE support comes as a side effect of having a thorough Linux compatibility layer rather than

"NVIDIA's Linux driver situation is complicated:

        **NVIDIA has two Linux drivers:**

        - **Proprietary driver** — the good one that everyone uses. High performance, full feature support. But it's a
        closed-source binary blob that's tightly coupled to the Linux kernel. You can't port it — you don't have the
        source code, and it depends on Linux kernel internals that change every release. NVIDIA would have to port it to
        your OS themselves, which they won't do for a new OS.

        - **Nouveau** (open source, reverse-engineered) — historically terrible performance because NVIDIA didn't
        release documentation or firmware. However, this is changing: NVIDIA has been releasing open-source kernel
        modules since 2022 (for Turing and newer GPUs, GTX 1600+/RTX 2000+). The open-source driver is improving but
        still significantly behind the proprietary one, especially for gaming and compute.

        **AMD is much easier:**

        AMD's Linux GPU driver (AMDGPU) is fully open source, included in the Linux kernel, and well-documented. It
        supports all modern AMD GPUs with good performance. This is why most new OS projects, Linux distros, and even
        the Steam Deck chose AMD.

        **What to do:**

        1. **Start with AMD** — port the AMDGPU driver. Open source, well-documented, good performance, covers a large
        chunk of desktop users.
        2. **Intel integrated graphics** — also fully open source (i915/xe driver). Covers most laptops and desktops
        that don't have a discrete GPU.
        3. **NVIDIA later** — wait for their open-source driver to mature, then port that. Or hope that your Linux
        syscall compatibility layer eventually becomes good enough to run their proprietary driver's userspace
        components against your kernel's DRM/KMS implementation. That's how FreeBSD handles NVIDIA — their Linux
        compatibility layer runs the Linux NVIDIA driver.

        The FreeBSD approach is actually the most promising path for NVIDIA support on your OS."
